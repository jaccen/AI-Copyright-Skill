# AI知识产权文件生成

面向**AI研发与AI行业应用**的知识产权文件生成技能，覆盖三大产出路径：

- **专利路径**：从AI项目资料生成符合国知局格式的发明专利申请文件（权利要求书、说明书、摘要），含技术交底书中间产物
- **软著路径**：从AI项目代码生成符合CPCC登记要求的软件著作权登记材料（说明书、源代码文档）
- **交底书路径**：从论文/研究笔记生成可交付给代理人的技术交底书

三条路径共享需求诊断与资料解读阶段，生成阶段各自独立，均内建AI领域专业规则。

## 触发条件

用户提及以下任一场景时启用：

- 专利申请、权利要求书、说明书、专利撰写、发明专利、实用新型、AI专利
- 软著、软著登记、软件著作权、软件说明书、源代码文档
- 技术交底书、专利交底书、技术披露书
- IP申请、知识产权申报、知产材料、AI知识产权
- 论文转专利、paper to patent
- 斜杠指令：`/ai-copyright`、`/AI知产`

**迭代意图**：当用户在已有产出上继续修改（改权利要求、补实施例、调整说明书等），直接进入对应路径的迭代修正流程，无需重新走完整流水线。

## 总体流程

```
Phase 0 可专利性预判（仅专利路径）
  └─ AI领域可专利性三要素检测
Phase A 需求诊断
  ├─ 确定路径：专利 / 软著 / 交底书 / 组合
  └─ 收集AI项目类型与边界信息
Phase B 资料解读与项目识别
  ├─ 自动识别AI项目类型
  └─ 提取技术要点，形成要点清单
Phase C 生成（按路径分支）
  ├─ C1 专利申请文件
  │   ├─ C1.1 现有技术检索与差异化
  │   ├─ C1.2 专利布局建议（单件/分案）
  │   ├─ C1.3 技术交底书（中间产物，用于交付代理人）
  │   ├─ C1.4 权利要求书
  │   ├─ C1.5 说明书
  │   ├─ C1.6 说明书摘要
  │   └─ C1.7 量化自检
  ├─ C2 软著登记材料
  │   ├─ C2.1 AI软件说明书
  │   ├─ C2.2 源代码文档
  │   └─ C2.3 量化自检
  └─ C3 技术交底书（独立路径）
      ├─ C3.1 论文/笔记→交底书映射
      ├─ C3.2 交底书撰写
      └─ C3.3 量化自检
Phase D 确认关卡（每阶段结尾）
Phase E 迭代修正
Phase F Word文档输出（默认自动执行）
Phase G 简介PPT输出（专利路径默认，其他路径可选）
```

## Phase 0 可专利性预判

**触发条件**：用户选择专利路径时自动执行。

依据 `references/ai-patent-special.md` 第1节，检测三要素：

| 要素 | 检测内容 | 通过标准 |
|------|----------|----------|
| 技术问题 | 是否锚定具体技术场景 | 非"效率低""精度差"等抽象表述 |
| 技术手段 | 算法步骤是否与系统架构绑定 | 每一步骤关联硬件/软件组件 |
| 技术效果 | 是否可量化 | 有具体数值或对比基准 |

**判定结果输出**：
- 三要素全过 → 继续专利路径
- 技术/技术手段不过 → 转交底书路径（积累完善后转专利）
- 技术效果不过 → 继续专利路径，但需在说明书中补充定量对比

## Phase A 需求诊断

向用户确认以下信息（不全时可跳过，后续推断并标注假设）：

1. **路径选择**：专利申请 / 软著登记 / 交底书 / 组合？
2. **技术主题**：一句话描述AI技术领域（如"基于3DGS的大场景实时重建"）
3. **AI项目类型**（自动检测辅助）：
   - 模型架构创新 / 训练方法创新 / 推理优化 / 数据处理 / 多模态融合 / Agent应用
4. **申请人信息**：名称、地址（脱敏时用占位符）
5. **发明人**：姓名（脱敏时用占位符）
6. **已有材料**：代码仓库 / 论文 / 设计文档 / 研究笔记？

**确认关卡**：输出3-5行诊断摘要，等待用户确认路径与边界后再进入Phase B。

## Phase B 资料解读与项目识别

### B.1 AI项目自动识别

按以下优先级扫描项目结构，依据 `references/ai-software-copyright-guide.md` 第1.1节决策树：

```
Read项目目录 → 识别入口文件类型 → 匹配AI项目类型
├── app.py/main.py/serve.py → AI服务类
├── train.py/trainer.py → AI训练类
├── inference.py/predict.py → AI推理类
├── pipeline.py/workflow.py → AI流水线类
└── package.json → 前端/全栈类
```

### B.2 技术要点提取

Read项目目录，按以下优先级提取：

| 优先级 | 来源 | 关注内容 |
|--------|------|----------|
| 1 | 模型定义文件 | 网络架构、创新模块、损失函数 |
| 2 | 训练/推理脚本 | 算法流程、训练策略、优化方法 |
| 3 | 论文/报告 | 方法描述、实验结果、技术对比 |
| 4 | 设计文档/架构文档 | 系统架构、模块划分、核心流程 |
| 5 | README/文档 | 功能说明、使用方式 |

提取结果形成**技术要点清单**，包含：
- 核心创新点（区分：架构级/算法级/工程级）
- 技术方案骨架
- 关键参数/阈值
- 与现有技术的区别
- 可量化技术效果

**确认关卡**：展示技术要点清单，等待用户确认或补充。

## Phase C1 专利申请文件生成

### C1.1 现有技术检索与差异化

使用联网搜索工具检索相关专利和论文：
- 搜索关键词从技术要点清单中提炼，分2-3轮搜索
- 搜索范围：国知局专利库、Google Patents、arXiv相关论文
- 每条现有技术记录：来源标识、技术方案概要、局限性
- 总结本发明与现有技术的**本质区别**
- 建议IPC/CPC分类（参考 `references/ai-patent-special.md` 第6节）

### C1.2 专利布局建议

依据 `references/ai-patent-special.md` 第2节，向用户建议：

| 创新分布 | 布局策略 |
|----------|----------|
| 核心创新在模型架构 | 单件涵盖架构+训练+推理 |
| 架构+训练各有独立创新点 | 分2件（架构件+训练件） |
| 含独立部署优化创新 | 分3件（架构+训练+部署） |

输出布局建议后等待用户确认。

### C1.3 技术交底书（中间产物）

**定位**：专利路径的中间交付物，用于提交给专利代理人。交底书记录技术要点、创新逻辑与实施方式，是权利要求书和说明书的撰写基础。

**数据来源**：复用 Phase B 提取的技术要点清单 + C1.1 检索结果 + C1.2 布局建议。

**章节结构**：

```
一、发明名称
二、技术领域
三、背景技术与现有技术缺陷
  3.1 现有技术概况（引用C1.1检索结果）
  3.2 现有技术存在的问题（指向本发明要解决的痛点）
四、发明内容
  4.1 发明目的（要解决的技术问题）
  4.2 技术方案（核心创新+完整流程，基于技术要点清单展开）
  4.3 技术效果（量化对比）
五、附图说明
  5.1 系统架构图
  5.2 核心流程图
  5.3 数据流示意图
六、具体实施方式
  6.1 实施例一（最优实施方式）
  6.2 实施例二（替代方案，来源自消融/变体）
七、关键创新点摘要（3-5条，用于代理人快速把握）
```

**与C3独立路径的区别**：

| 维度 | C1.3（专利路径中间产物） | C3（独立交底书路径） |
|------|--------------------------|----------------------|
| 输入 | 项目代码+技术要点清单 | 论文/研究笔记 |
| 映射方式 | 技术要点→交底书结构 | 论文6类要素→交底书 |
| 目的 | 供代理人撰写正式申请 | 直接交付交底书 |
| 后续 | 继续C1.4权利要求书 | 无后续生成 |

**确认关卡**：展示交底书摘要，等待用户确认技术要点无遗漏后再进入C1.4。

### C1.4 权利要求书

撰写规范见 `references/ai-patent-claims-guide.md`。

**核心结构**：
- **方法权利要求**（独立1项 + 从属3-8项）
- **系统/装置权利要求**（独立1项 + 从属3-8项，与方法步骤一一对应）
- **计算机可读存储介质权利要求**（独立1项）

**AI领域特殊要求**（依据 `references/ai-patent-special.md` 第1.3节）：
1. 每一步骤需关联系统组件（"通过GPU并行计算单元执行XX算子"）
2. 模型架构类：按"子模块→处理动作→中间结果"链式展开
3. 训练方法类：需包含训练数据构建、损失函数、优化策略
4. 推理优化类：需包含原始模型→优化操作→推理策略三段
5. 多模态类：需包含各模态编码→跨模态对齐→融合→解码
6. Agent类：需包含意图识别→工具选择→执行→响应生成

**从属权利要求展开策略**（5层递进）：
1. 模块内部结构细化
2. 具体计算方式/公式
3. 参数取值范围
4. 优选实施方式
5. 训练相关特征

**确认关卡**：展示权利要求书草案，等待用户确认保护范围。

### C1.5 说明书

**章节结构**：

```
一、技术领域
二、背景技术
  2.1 现有技术描述（含检索结果）
  2.2 现有技术缺陷
三、发明内容
  3.1 要解决的技术问题
  3.2 技术方案（对应权利要求逐项展开）
  3.3 有益效果（必须量化：具体数值+对比基准）
四、附图说明
  （AI系统附图需求见 references/ai-patent-special.md 第4节）
五、具体实施方式
  5.1 实施例一（对应独立权利要求）
  5.2 实施例二（训练方法实施方式）
  5.3 实施例三（推理部署实施方式）
```

**AI领域脱敏规则**（依据 `references/ai-patent-special.md` 第5节）：
- 具体数据集名 → "预设道路场景数据集"
- 模型参数量 → "预设规模的预训练模型"
- 硬件型号(A100/H100) → "图形处理器"
- 开源框架名 → "深度学习框架"
- 训练成本/时长 → "预定训练周期"

**附图**：系统框图和流程图使用fenced mermaid（`flowchart TB`/`LR` + `subgraph`），在说明书中用"图X"引用并配文字说明。AI架构图必须有数据流动路径标注。

### C1.6 说明书摘要

- 300字以内
- 涵盖：技术领域 + 核心技术方案 + 主要技术效果
- 不含商业性宣传用语
- 算法名称需用通用表述替换

### C1.7 专利量化自检

生成后内部执行（不写入交付文件），采用100分制评分：

| 检查项 | 分值 | 评分标准 |
|--------|------|----------|
| 独立权利要求必要技术特征完整性 | 15 | 每缺一项扣5分 |
| 从属权利要求回引正确性 | 10 | 每处错误扣3分 |
| 方法+系统+介质三件套完整 | 10 | 每缺一类扣5分 |
| 说明书充分公开 | 15 | 本领域技术人员可实施 |
| 实施方式覆盖全部权利要求特征 | 15 | 每未覆盖特征扣3分 |
| 有益效果量化支撑 | 10 | 空泛断言每处扣3分 |
| 全文术语统一性 | 5 | 同义不同名每处扣1分 |
| 摘要与权利要求1对应 | 5 | 不对应扣5分 |
| 脱敏完整性 | 10 | 每处泄露扣3分 |
| 附图编号与正文引用一致性 | 5 | 每处不一致扣1分 |

**80分以上**：直接交付。**60-80分**：自动修正后交付并标注修正点。**60分以下**：返回Phase C1重写并说明不足。

## Phase C2 软著登记材料生成

软著登记材料规范见 `references/ai-software-copyright-guide.md`。

### C2.1 AI软件说明书

**格式要求**：10-15页，A4排版，含截图≥6张。

**AI项目专用章节**（依据 `references/ai-software-copyright-guide.md` 第3节）：

```
1. 引言
   1.1 编写目的
   1.2 软件概述（含AI能力描述，面向审查员，避免专业术语）
   1.3 运行环境（含GPU/驱动/CUDA要求）
2. 安装与初始化
   2.1 环境配置（Python版本+深度学习框架版本+GPU驱动）
   2.2 依赖安装
   2.3 模型权重获取
   2.4 配置说明
3. 功能说明
   3.1 功能模块总览
   3.2 AI核心引擎（模型加载→推理→后处理）
   3.3 数据处理模块
   3.4 服务接口/API模块
   3.5 管理与监控模块（如有）
4. 非功能需求
5. 常见问题
```

**撰写注意**：
- 语言面向审查员，避免"fine-tuning""inference"等英文行话
- 截图用 `[截图: 功能名称界面]` 占位
- 即使是服务端程序也需描述部署/配置/监控过程（软著要求"人机交互"）
- 开源预训练权重在声明中注明不在保护范围

### C2.2 源代码文档

**格式要求**：前30页+后30页，每页≥50行。

**AI项目源文件优先级**（依据 `references/ai-software-copyright-guide.md` 第2节）：

| 优先级 | 文件 | 必选/可选 |
|--------|------|-----------|
| 1 | model.py/network.py | 必选 |
| 2 | train.py/trainer.py | 必选 |
| 3 | inference.py/predict.py | 必选 |
| 4 | dataset.py/dataloader.py | 推荐 |
| 5 | loss.py/criterion.py | 推荐 |
| 6 | config.yaml/hparams.py | 可选 |

**生成步骤**：
1. Grep/Glob扫描项目源代码目录
2. 按优先级排序，拼接代码
3. 格式化为每页50行
4. 总行数 > 3000行时取前1500行 + 后1500行
5. 页眉添加软件名称+版本号

**脱敏清理**（执行 `references/ai-software-copyright-guide.md` 第2.2节清单）：
- 删除所有API Key/Token/Secret
- 删除绝对路径和内网地址
- 删除个人邮箱/姓名
- 保留算法说明注释

### C2.3 软著量化自检

| 检查项 | 分值 | 评分标准 |
|--------|------|----------|
| 说明书页数10-15页 | 15 | 不足10页扣10分，超15页扣5分 |
| 截图占位≥6处 | 10 | 每少1处扣2分 |
| 功能覆盖完整 | 15 | 每遗漏核心功能扣5分 |
| AI能力描述面向非技术人员 | 10 | 每处专业术语未解释扣2分 |
| 环境要求含GPU/CUDA信息 | 10 | 缺失扣10分 |
| 源代码文档页数达标 | 15 | 不达标扣10分 |
| 源代码每页≥50行 | 10 | 不达标扣5分 |
| 名称版本号一致性 | 5 | 不一致扣5分 |
| 无商业秘密泄露 | 10 | 每处泄露扣5分 |

## Phase C3 技术交底书生成

### C3.1 论文/笔记→交底书映射

依据 `references/ai-patent-special.md` 第3节6类映射表：

| 输入要素 | 交底书映射 | 转换动作 |
|----------|------------|----------|
| Research Problem | 技术问题 | 学术语言→工程痛点 |
| Contribution | 技术方案 | "we propose"→步骤化描述 |
| Method Modules | 实施单元 | 每module→单元+步骤 |
| Algorithm Flow | 实施流程 | 伪代码→步骤+流程图 |
| Experimental Outcome | 预期效果 | 定量→"相比XX提升YY%" |
| Ablation Study | 备选方案 | 变体→备选实施方式 |

### C3.2 交底书撰写

**章节结构**：

```
一、发明名称
二、技术领域
三、背景技术与现有技术缺陷
  3.1 现有技术概况
  3.2 现有技术存在的问题（指向本发明要解决的痛点）
四、发明内容
  4.1 发明目的（要解决的技术问题）
  4.2 技术方案（核心创新+完整流程）
  4.3 技术效果（量化对比）
五、附图说明
  5.1 系统架构图
  5.2 核心流程图
  5.3 数据流示意图
六、具体实施方式
  6.1 实施例一（最优实施方式）
  6.2 实施例二（替代方案）
七、关键创新点摘要（3-5条，用于代理人快速把握）
```

### C3.3 交底书量化自检

| 检查项 | 分值 | 评分标准 |
|--------|------|----------|

| 材料完整性 | 10 | 缺章节扣5分 |

## Phase D 确认关卡

每个Phase结束后，向用户展示当前阶段产出的**摘要**，并明确以下选项：

1. **确认继续** → 进入下一阶段
2. **修改当前阶段** → 说明修改点，原地修正后重新展示
3. **切换路径** → 从当前阶段跳转到另一路径的对应阶段
4. **暂停存档** → 保存当前进度到 `.temp/ai-ip-progress.md`

## Phase E 迭代修正

当用户在已有产出上继续修改时：

1. **识别改什么**：权利要求调整 / 说明书补充 / 软著修改 / 交底书修改
2. **定位段落**：Read已交付文件，定位需修改部分
3. **定向修正**：仅修改指定部分，保持其余内容不变
4. **差异标注**：在修改处以 `<!-- 修订: 原文X→新文Y -->` 标注变更
5. **另存交付**：新版本以 `{案件名}_v{N}.md` 命名，不覆盖旧版
6. **重跑自检**：仅对修改涉及的自检项重新评分

迭代时禁止：重跑完整流水线（除非用户明确要求）；修改未提及的章节。

## Phase F Word文档输出

**触发条件**：Phase C生成完成且自检通过后自动执行（用户明确要求Markdown时除外）。

### F.1 输出策略

| 路径 | 默认输出 | 说明 |
|------|----------|------|
| 专利 | 3个文件 | 交底书.docx + 专利.docx（封面+权利要求+说明书+摘要）+ 简介.pptx |
| 软著 | 2个 `.docx` 文件 | 软件说明书.docx + 源代码文档.docx |
| 交底书 | 单个 `.docx` 文件 | 含完整交底书全部章节 |
| 组合 | 多个文件 | 按路径分别输出 |

### F.2 Word文档生成流程

使用内置 `docx` 技能的 docx-js 工作流：

1. **构建JS脚本**：在 `.temp/` 下生成 `{案件名}-docx.js` 脚本
2. **脚本内容**：
   - 引用 `docx` 包（Document, Packer, Paragraph, TextRun, ...）
   - 按路径构建对应的section结构（封面/权利要求书/说明书/摘要等）
   - 专利文档格式规范：
     - 字体：正文宋体(size:24)、标题黑体(size:32/28)、封面标题黑体(size:36)
     - 行距：1.5倍（line:360）
     - 首行缩进：正文段落缩进480DXA
     - 页边距：上下左右各1440DXA（1英寸）
     - 页眉：section名称（如"权利要求书""说明书"）
     - 页脚：页码（第X页）
   - 软著文档格式规范：
     - 软件说明书：宋体正文、含截图占位标记、A4排版
     - 源代码文档：等宽字体(Courier New, size:18)、每页50行、页眉含软件名+版本号
3. **执行生成**：`node .temp/{案件名}-docx.js`
4. **验证输出**：检查文件大小 > 5KB，确认未损坏

### F.3 专利Word文档结构

```
文件1: {案件名}_交底书.docx（C1.3中间产物）
  Section 1: 封面（发明名称+技术领域）
  Section 2-7: 交底书一~七章
  - 正文宋体(size:24)、标题黑体(size:28)、首行缩进480DXA
  - 页眉"技术交底书"

文件2: {案件名}.docx（正式申请文件）
  Section 1: 封面
    - 发明专利申请（居中，黑体36pt加粗）
    - 发明名称（居中，黑体36pt加粗）
    - 申请人/发明人/申请日期占位符

  Section 2: 权利要求书
    - 页眉"权利要求书"
    - 页脚页码
    - 每项权利要求独立段落，权利要求编号加粗
    - 步骤S1-S6缩进排列

  Section 3: 说明书
    - 页眉"说明书"
    - 页脚页码
    - 一~五章标题黑体加粗
    - 实施方式段落首行缩进

  Section 4: 说明书摘要
    - 页眉"说明书摘要"
    - 单段文本，300字以内
```

### F.4 软著Word文档结构

```
文件1: {软件名}_软件说明书.docx
  - 10-15页A4排版
  - 截图占位标记 [截图: 功能名称界面]
  - 章节1-5完整

文件2: {软件名}_源代码文档.docx
  - 等宽字体Courier New
  - 页眉含软件名称+版本号
  - 前30页+后30页，每页≥50行
```

### F.5 备选输出格式

用户可指定以下格式替代默认Word输出：

- `md`：仅输出Markdown（不执行Phase F）
- `docx`：输出Word（默认）
- `both`：同时输出Markdown和Word

## Phase G 简介PPT输出

**触发条件**：专利路径完成后默认执行；其他路径用户可指定 `--ppt` 触发。

### G.1 PPT定位

简介PPT用于向管理层/评审委员会/合作方快速介绍发明方案，不是正式申请文件，而是**内部汇报与沟通工具**。一页一个核心信息，5-8页即可讲清楚。

### G.2 PPT页面结构

```
第1页 封面
  - 发明名称（大字居中）
  - 申请人 / 日期
  - 可选：技术领域标签

第2页 技术背景与痛点
  - 现有技术的1-3个核心缺陷
  - 痛点量化（如"推理延迟120ms，无法满足实时需求"）
  - 布局：左侧文字 + 右侧配图占位

第3页 核心创新方案
  - 本发明的技术方案骨架（3-5个步骤/模块）
  - 创新点1-2-3逐条列出
  - 布局：流程图或列表

第4页 系统架构
  - 系统框图（与交底书/说明书附图对应）
  - 核心模块标注创新点

第5页 技术效果对比
  - 量化效果表格（指标 / 现有方案 / 本发明 / 提升%）
  - 可选：柱状图或雷达图

第6页 专利布局（如有多件）
  - 系列申请关系图
  - 或单件申请的权利要求保护范围概览

第7页 实施计划与下一步
  - 专利申请时间线
  - 已有成果（论文/代码/产品）
  - 后续计划

第8页 附录（可选）
  - 关键公式 / 实验数据详表 / 权利要求摘要
```

### G.3 PPT生成流程

使用内置 `pptx` 技能的 python-pptx 工作流：

1. **构建Python脚本**：在 `.temp/` 下生成 `{案件名}-ppt.py` 脚本
2. **脚本内容**：
   - 引用 `pptx` 包（Presentation, Slide, TextFrame, ...）
   - 按G.2页面结构逐页构建
   - PPT格式规范：
     - 16:9宽屏比例
     - 主题色：科技蓝（#1B3A5C）+ 亮蓝（#3B82F6）+ 白色
     - 标题字体：微软雅黑(Microsoft YaHei)，32pt加粗
     - 正文字体：微软雅黑，18pt
     - 表格/图表使用pptx内置Table/Chart
     - 配图占位：`[图: 描述]` 文字占位
3. **数据来源**：
   - 第2页 ← C1.1检索结果 + C1.3交底书第三章
   - 第3页 ← C1.3交底书第四章
   - 第4页 ← C1.3交底书第五章 / C1.5说明书附图
   - 第5页 ← Phase B技术要点清单的量化效果
   - 第6页 ← C1.2布局建议
4. **执行生成**：`python .temp/{案件名}-ppt.py`
5. **验证输出**：检查文件大小 > 50KB，页面数5-8页

### G.4 PPT输出命名

```
outputs/{案件标识}/
└── patent/
    └── {案件名}_简介.pptx
```

## 输出交付规范

### 文件命名

```
outputs/{案件标识}/
├── patent/
│   ├── {案件名}_disclosure.md           ← 交底书Markdown（C1.3中间产物）
│   ├── {案件名}_disclosure.docx         ← 交底书Word
│   ├── {案件名}_claims.md
│   ├── {案件名}_specification.md
│   ├── {案件名}_abstract.md
│   ├── {案件名}_full.md
│   ├── {案件名}.docx                    ← 专利Word完整文件
│   └── {案件名}_简介.pptx               ← 简介PPT
├── software-copyright/
│   ├── {软件名}_manual.md
│   ├── {软件名}_manual.docx             ← 软著说明书Word
│   ├── {软件名}_source_code.md
│   └── {软件名}_source_code.docx        ← 源代码文档Word
└── disclosure/
    ├── {案件名}_disclosure.md
    └── {案件名}_disclosure.docx          ← 交底书Word
```

### 格式

- **默认输出Word(.docx)格式**，同时保留Markdown中间文件
- Word文档通过内置 `docx` 技能的 docx-js 工作流生成
- PPT通过内置 `pptx` 技能的 python-pptx 工作流生成
- mermaid图在Markdown中保留源码；Word中转为文字描述或用diagram-drawing技能渲染PNG后嵌入
- 量化自检报告单独输出为 `{案件名}_selfcheck.md`

### 禁止事项

- 交付文件中不得出现skill名、仓库路径、免责脚注
- 正文不得包含自检清单章节
- 不得虚构专利号、链接或检索结果
- 权利要求书中不得出现"大约""左右"等不确定用语
- 摘要不得包含商业性术语（"颠覆性""领先""首创"等）
