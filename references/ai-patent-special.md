# AI领域专利撰写特殊指南

## 1. 算法可专利性判定框架

### 1.1 中国国知局审查标准（2026版审查指南第九章第6节）

| 判定层级 | 通过条件 | 常见驳回理由 |
|----------|----------|--------------|
| 第一层：不属于智力活动的规则和方法 | 算法与技术手段结合解决技术问题；需通过专利法第五条第一款伦理审查（数据采集/标签管理/规则设置/推荐决策不得违反法律、社会公德或妨害公共利益） | 纯数学优化、业务流程自动化 |
| 第二层：具备技术特征 | 产生技术效果（速度提升、精度提高、资源节约） | 仅提高计算效率但无技术场景 |
| 第三层：充分的说明书公开 | 公开训练数据构成、模型结构、推理流程至可实施；说明书需公开模型必要模块、层级、连接关系、训练必需步骤与参数；模型应用于具体领域时需公开模型与领域结合方式、输入输出内在关联关系 | "黑箱"型公开不充分 |

### 1.2 AI专利撰写三要素

1. **技术问题**：必须锚定具体技术场景（如"三维场景重建中的实时渲染瓶颈"），不可是"效率低""精度差"等抽象表述
2. **技术手段**：将算法步骤与硬件/软件架构绑定（如"GPU并行计算单元执行XX算子"），每一算法步骤需关联系统组件
3. **技术效果**：量化技术效果（如"推理延迟从120ms降至35ms"），优于"效果显著提升"

### 1.3 各领域可专利性风险与对策

| AI领域 | 典型风险 | 撰写对策 |
|--------|----------|----------|
| 生成式AI（文生图/视频/3D） | "纯内容生成"被认定为智力活动规则 | 必须绑定具体技术场景（如"工业设计稿自动生成""医学影像增强"），权利要求需体现"条件控制→模型推理→技术输出"而非仅"输入文本→输出图像" |
| AIGC水印与版权保护 | "纯信息标记"被认定为信息表达方法 | 需将水印嵌入与模型内部层绑定（如"在去噪网络的第K层注入隐式标记"），需量化鲁棒性指标 |
| 具身智能 | "纯运动控制规划"被认定为智力活动规则 | 每一步骤需绑定传感器输入和执行器输出（如"基于深度相机获取的点云数据，通过力矩控制器驱动关节执行抓取"） |
| 强化学习 | "纯策略优化"落入数学方法 | 需绑定物理系统（如"用于机器人路径规划的强化学习方法"，奖励函数需包含物理约束项） |
| RAG/检索增强生成 | "纯信息检索"被认定为信息表达方法 | 需体现"查询向量化→语义匹配→上下文重构→生成约束"的技术链路，而非仅"检索→输出" |
| NLP/大语言模型 | "纯语言处理"落入智力活动规则 | 需绑定具体技术场景（如"用于代码生成的""用于多模态对齐的"），步骤需关联计算组件 |
| 金融风控 | "纯金融策略"落入商业方法 | 需绑定数据处理手段（如"基于图神经网络的交易异常检测"），量化检测指标 |
| AI对齐/可解释性 | "纯评估方法"落入智力活动规则 | 需绑定"对齐→特定应用场景的安全保障效果"（如"用于自动驾驶决策的安全对齐方法"） |
| 世界模型(World Model) | "纯视频预测"被认定为智力活动规则 | 必须绑定具体技术场景（如"用于自动驾驶仿真的世界模型"），需体现物理一致性约束和状态转移的可解释性 |
| SSM/Mamba/状态空间模型 | "纯序列建模"落入数学方法 | 需绑定具体技术场景（如"用于实时语音识别的状态空间模型"），需体现选择性扫描/门控机制与系统组件的绑定 |
| 大模型推理优化(MoE/量化/蒸馏) | "纯模型压缩"落入数学方法 | 需绑定具体部署场景（如"端侧NPU推理优化"），量化蒸馏需体现硬件感知和精度保持的技术效果 |
| AI辅助发明(AI as tool) | AI不能列为发明人，人类贡献需明确 | 发明人必须是自然人；需明确人类在技术方案选择、参数调整、效果验证等环节的创造性贡献 |
| 多Agent协作系统 | "纯信息交互"落入智力活动规则 | 需体现Agent间通信协议与系统架构的绑定，每个Agent需关联具体功能模块 |

### 1.4 独立权利要求AI通用模板

```
一种基于[技术场景]的[方法名称]方法，其特征在于，包括：

S1. [数据获取/预处理步骤]，通过[具体技术手段]获取[输入数据描述]；
S2. [核心算法步骤1]，利用[模型/网络名称]对所述[输入数据]进行[处理动作]，
    所述[模型/网络]包括[结构特征]，以[技术目的]；
S3. [核心算法步骤2]，基于S2的输出，通过[具体计算方式]确定[中间结果]；
S4. [输出/应用步骤]，根据所述[中间结果]生成[最终输出]，用于[技术应用场景]。

其中，所述[模型/网络]的训练过程包括：
T1. [训练数据构建步骤]；
T2. [损失函数与优化步骤]；
T3. [训练收敛条件]。
```

## 2. AI技术方案拆分布局策略

### 2.1 单发明vs分案申请判断

| 判断维度 | 单发明（一件申请） | 分案申请（多件） |
|----------|---------------------|-------------------|
| 模型架构+训练方法 | 核心创新在架构，训练方法是配套 | 架构和训练各有独立创新点 |
| 推理方法+部署方案 | 推理优化是架构的自然延伸 | 部署方案引入新的硬件/编排创新 |
| 数据处理+下游应用 | 应用是数据处理流程的终端 | 应用场景差异大（如医疗vs工业） |
| 3D重建+渲染优化 | 渲染优化是重建方法的延伸 | 渲染引入新表示/管线创新 |
| 生成模型+编辑控制 | 控制机制是生成模型的配套 | 控制机制是独立的条件注入架构 |
| 感知+决策+执行（具身智能） | 三者为同一闭环的组件 | 感知模型与决策策略各自独立创新 |

### 2.2 推荐布局：3件系列申请

1. **核心架构/算法件**：保护模型结构、核心算法流程、损失函数设计
2. **训练方法件**：保护数据构建、采样策略、训练技巧、超参数搜索
3. **推理部署件**：保护模型压缩、量化策略、推理优化、服务架构

### 2.3 具身智能布局特例

具身智能项目若同时包含感知（视觉-语言-动作模型）和执行（控制策略），建议2件布局：
1. **感知决策件**：保护VLA模型架构、多模态感知融合、策略网络
2. **控制执行件**：保护动作空间设计、力控策略、反馈闭环机制

## 3. 论文到专利映射表（扩展型）

### 3.1 通用6类映射

| 论文要素 | 专利映射 | 转换要点 |
|----------|----------|----------|
| Research Problem | 技术问题 | 从学术语言转为工程痛点描述 |
| Contribution | 技术方案 | 将"we propose"转为"包括...步骤" |
| Method Modules | 装置单元 | 每个module对应一个装置单元+方法步骤 |
| Algorithm Flow | 实施流程 | 伪代码→步骤化描述，附流程图 |
| Experimental Outcome | 技术效果 | 定量结果→"进一步地，本发明相比XX方法提升YY%" |
| Ablation Study | 从属权利要求 | 每个ablation变体→一条从属权利要求 |

### 3.2 特定领域映射补充

| 领域 | 论文特有要素 | 额外专利映射 |
|------|-------------|-------------|
| 3D视觉 | 渲染公式（体渲染/EWA splatting） | 映射为权利要求中的"渲染计算步骤"，公式参数展开为从属权利要求 |
| 3D视觉 | 相机标定/畸变模型 | 映射为"数据预处理步骤"中的标定子步骤 |
| 生成式AI | 采样调度/噪声策略 | 映射为"噪声调度步骤"，不同调度策略→从属权利要求 |
| 生成式AI | 条件控制方式（ControlNet/IP-Adapter） | 映射为"条件注入步骤"，每种控制模态→从属权利要求 |
| 具身智能 | Sim2Real迁移策略 | 映射为"从仿真环境到实物的参数适配步骤" |
| 具身智能 | 动作空间定义 | 映射为"动作编码步骤"，连续/离散动作→实施例变体 |
| 强化学习 | 奖励函数设计 | 映射为"奖励计算步骤"，物理约束项需展开 |
| RAG | 检索策略对比 | 映射为"混合检索步骤"，不同检索器→从属权利要求 |
| AI水印 | 攻击鲁棒性实验 | 映射为"水印验证步骤"，各攻击类型→从属权利要求 |
| AI4Science | 物理守恒约束 | 映射为"约束损失计算步骤"，守恒律→必要性技术特征 |

## 8. 2026版审查指南AI专项审查规则（第84号令）

### 8.1 AI伦理审查（专利法第五条第一款）

2025年11月公布第84号令，新增"6.1.1"条款：涉及AI/大数据的专利申请，若数据采集、标签管理、规则设置、推荐决策含有违反法律、社会公德或妨害公共利益的内容，不予授权。

审查对象扩展至说明书内容（不仅限权利要求）。

**两个官方审查示例**：
- 【例1】商场床垫精准营销系统：未获顾客单独同意采集人脸→违反《个人信息保护法》
- 【例2】无人驾驶应急决策模型：按性别/年龄差异化决策→违反社会公德（算法歧视）

**撰写对策**：
- 数据采集步骤中增加"经授权许可/去标识化处理"表述
- 决策模型涉及人的分类时增加"平等性约束"或"公平性正则化"步骤
- 推荐系统增加"用户可控的隐私保护设置"从属权利要求

### 8.2 AI创造性审查新标准

2026版指南新增三条关键审查原则：

**原则1：场景不同但算法实质相同→无创造性**
- 例：识别船上果实数量→改为识别船只数量，若算法/模型/训练流程无实质改变→无创造性
- 对策：新场景必须伴随算法层面的实质性调整，并将调整点写入独立权利要求

**原则2：场景相同但算法实质不同→有创造性**
- 例：废钢种类识别→改为废钢等级划分，调整了特征提取（颜色/边缘/纹理）、卷积层/池化层设置→有创造性
- 启示：强调"技术问题→技术手段→技术效果"三者的关联链条完整性

**原则3：未贡献的特征不计入创造性**
- 2026版明确：权利要求中对技术问题解决无贡献的特征，通常不会带来创造性
- 对策：独立权利要求中每个特征都应与解决技术问题有明确因果关系

### 8.3 AI说明书充分公开新标准

2026版指南新增AI模型公开要求（"6.3.1"及"6.3.3审查示例"）：

**模型构建/训练类**：必须公开模型必要模块、层级或连接关系、训练必需的具体步骤和参数

**模型具体应用类**：必须公开模型/算法如何与具体领域/场景结合、输入输出数据如何设置以表明内在关联关系

**公知常识例外**（例20）：若未公开的技术手段属于公知常识，不导致公开不充分

**非公知常识→公开不充分**（例21）：血常规+人脸图像预测癌症，但未公开哪些指标与癌症有关→公开不充分

**撰写对策**：
- 在说明书中增加"模型架构详述"章节，逐层描述网络结构
- 在实施例中增加"模型输入输出配置"小节，详述输入特征与输出结果的映射关系
- 复杂多模态输入需解释特征选择的技术依据

### 8.4 AI发明人资格新规

- 发明人必须是自然人，不得填写AI名称或单位
- 身份信息必须真实，不得虚假
- AI辅助发明需明确人类创造性贡献的具体环节

### 9.2 创新点深度挖掘检查表

对每个识别出的创新点，追问以下问题以确定专利价值：

1. **问题唯一性**：该创新点解决的技术问题是否在现有文献中未被提出？
2. **方案独特性**：技术手段是否与现有方案有本质区别（非参数替换/场景替换）？
3. **效果可量化**：技术效果是否可用具体指标量化（精度/速度/资源/成本）？
4. **可实施性**：本领域技术人员能否根据说明书实施？
5. **保护宽度**：独立权利要求的概括是否既不过窄（容易规避）又不过宽（无效风险）？
6. **多维度布局**：该创新点是否可拆分为架构/训练/应用多个保护方向？

## 10. AI领域高频驳回模式与应对

### 10.1 十大高频驳回理由与应对策略

| 驳回理由 | 频次 | 典型审查意见 | 应对策略 |
|----------|------|-------------|----------|
| 属于智力活动规则 | ★★★★★ | "该方案本质是数学算法/商业方法" | 在独立权利要求中增加"通过[硬件/系统组件]执行"绑定表述 |
| 缺乏创造性 | ★★★★★ | "与对比文件1的区别仅在于应用场景不同" | 证明场景迁移导致算法层面的实质性调整（参考§8.2原则1） |
| 公开不充分 | ★★★★ | "说明书未公开模型内部结构/参数" | 补充模型层级、连接关系、训练参数等细节 |
| 权利要求得不到说明书支持 | ★★★★ | "权利要求概括过于宽泛" | 缩小参数范围，或增加从属权利要求提供具体实施方式 |

| 技术效果不可信 | ★★★ | "效果数据缺乏实验数据支撑" | 补充对比实验数据，量化技术效果 |
| 发明人资格问题 | ★★ | "AI被列为发明人/信息不真实" | 确保发明人为自然人且信息真实 |
| 单一性缺陷 | ★★ | "多项独立权利要求不属于一个发明构思" | 调整权利要求结构，或分案申请 |

### 10.2 驳回复审中"修改超范围"的规避

修改规则：不得超出原说明书和权利要求书记载的范围。

**安全修改策略**：
- 可以：缩小权利要求范围、补入从属权利要求特征、澄清模糊表述
- 不可以：增加原文件未记载的技术特征、改变发明实质、补充实验数据（除非可从原文件得出）

### 10.3 OA答复常用论证模板

针对"缺乏创造性"的三步回应：
1. 确定最接近的现有技术（D1）
2. 指出区别特征及实际解决的技术问题（区别于D1所解决的技术问题）
3. 论证区别特征在D1中没有技术启示（无动机将D1的特征组合到D1中）

## 12. AI专利撰写语言规范与术语标准

### 12.1 术语统一规范

| 禁止用语 | 推荐替换 | 说明 |
|----------|----------|------|
| 大约/左右 | "在[X,Y]范围内" | 权利要求不允许不确定用语 |
| 上述/前述 | "[具体模块/步骤名称]" | 避免指代模糊 |
| 特征图 | "中间特征表示/中间特征矩阵" | 术语规范化 |
| 预训练模型 | "预先训练的[模型类型]模型" | 保持实现无关性 |
| 过拟合 | "在训练数据集上表现优于验证数据集" | 面向审查员描述 |
| 超参数 | "训练控制参数" | 避免过度专业化 |
| 效果显著 | "相比现有技术提升XX%" | 量化替代 |
| 颠覆性/领先/首创 | "相较于现有技术实现了XX改进" | 禁止商业宣传语 |
| GPU | "图形处理器" | 保持实现无关性 |
| Transformer/ResNet | "[具体技术类型]网络架构" | 可保留具体名称但需附通用描述 |

### 12.2 权利要求用语规范

- 方法步骤：以"获取→处理→计算→输出"链条展开，每步限定输入输出
- 装置权利要求：与方法步骤严格一一对应
- 参数限定：优先使用闭区间[X,Y]而非开区间
- 功能性限定：仅当本领域技术人员可确定实施方式时使用
- 多个实施方式：使用"所述...包括[方式A]或[方式B]"并列

### 12.3 说明书用语规范

- 技术效果：每条效果需有对比数据支撑
- 实施例：至少包含独立权利要求的全部特征+1种从属权利要求组合
- 附图说明：每幅附图需说明各组成部分及其连接关系
- 公式：首次出现的数学符号需定义，公式后附文字解释

## 13. AI专利PCT国际申请注意事项

### 13.1 AI专利PCT申请关键差异

| 维度 | 中国国知局 | USPTO | EPO |
|------|-----------|-------|-----|
| AI可专利性 | 需绑定技术场景 | Section 101: 需实际应用 | 技术效果要求 |
| 软件专利 | 含技术特征可授权 | 不限制纯软件 | 需技术特征 |
| AI伦理 | 2026新增第五条审查 | 无直接对应 | 道德条款 |
| 充分公开 | 模型结构+参数需详述 | enablement/written description | 欧洲标准较严 |

### 13.2 PCT申请撰写建议

- 权利要求结构：方法+系统+介质+设备（美国需Device Claim）
- 说明书中增加"实施例的多种变形"为后续国家阶段修改留空间
- 避免在说明书中公开不必要的技术秘密

## 14. EPO/USPTO 差异化撰写要点

### 14.1 USPTO 差异化要点

| 维度 | USPTO要求 | 撰写对策 |
|------|-----------|----------|
| 可专利性(35 USC 101) | 需绑定实际应用(practical application)，不可仅抽象概念 | 每个算法步骤需关联具体技术效果，权利要求需体现"技术方案整体而非算法本身" |
| 新颖性(35 USC 102) | 单一对比文件(single reference)可驳回 | 确保至少一个技术特征未被任何单篇文献公开 |
| 创造性(35 USC 103) | Graham要素分析：范围+内容+差异→显而易见 | 强化"技术问题的非显而易见性"论述，引用意想不到的技术效果 |
| 充分公开(35 USC 112(a)) | enablement: 本领域技术人员能实施 | AI模型需公开到可复现级别（结构+关键参数+训练数据构成） |
| 明确性(35 USC 112(b)) | claims must be definite | 禁止"大约""左右"等模糊用语；"所述"指代必须唯一 |
| Device Claim | 美国专利要求独立的Device/Apparatus Claim | 在方法+系统+介质三件套基础上增加设备权利要求 |

### 14.2 EPO 差异化要点

| 维度 | EPO要求 | 撰写对策 |
|------|---------|----------|
| 技术性(Art. 52(1)) | 需产生"技术效果"(technical effect) | AI方法需证明对物理/工程参数的改进，而非仅信息处理效率 |
| 创造性(Art. 56) | "Problem-Solution Approach"：问题→方案→效果 | 先定义客观技术问题(不预设发明)，再论证技术方案的非显而易见性 |
| 独立权利要求(Art. 84) | 两段式写法：前序部分+特征部分("characterized in that") | 区别于中国"其特征在于"，EPO用"characterized in that" |
| 充分公开(Art. 83) | European standard: 需"repeat the invention" | 公开程度需使欧洲本领域技术人员无需过度负担即可实施 |
| 医疗AI例外 | Art. 53(c): 诊断方法不可专利 | 医疗AI需绑定"治疗方法改进"而非"疾病诊断" |

### 14.3 中/美/欧三局权利要求格式对照

```
中国国知局：
一种基于[场景]的[方法]方法，其特征在于，包括：
S1. ...；

USPTO：
A method for [method name], comprising:
S1. ...;

EPO：
A method for [method name], characterized in that the method comprises:
S1. ...;
```
